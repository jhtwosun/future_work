## Abstract (verbatim)

"Test-time scaling (TTS) has proven effective in enhancing the reasoning capabilities of large language models (LLMs). Verification plays a key role in TTS, simultaneously influencing (1) reasoning performance and (2) compute efficiency, due to the quality and computational cost of verification. In this work, we challenge the conventional paradigms of verification, and make the first attempt toward systematically investigating the impact of verification granularity-that is, how frequently the verifier is invoked during generation, beyond verifying only the final output or individual generation steps. To this end, we introduce Variable Granularity Search (VG-Search), a unified algorithm that generalizes beam search and Best-of-N sampling via a tunable granularity parameter g. Extensive experiments with VG-Search under varying compute budgets, generator-verifier configurations, and task attributes reveal that dynamically selecting g can improve the compute efficiency and scaling behavior. Building on these findings, we propose adaptive VG-Search strategies that achieve accuracy gains of up to 3.1% over Beam Search and 3.6% over Best-of-N, while reducing FLOPs by over 52%. We will open-source the code to support future research."

## Key idea (3-5 sentences)

The "step-level vs. solution-level vs. token-level" choice for verifier invocation is recast as a single tunable parameter *g* — the number of generated tokens (or sub-steps) between successive verifier calls. Setting g=1 recovers token-level verification, g=infinity recovers Best-of-N (verify only the final answer), intermediate g recovers beam-search-style step-level verification. The paper introduces **VG-Search**, a unified search algorithm parameterized by g, and empirically shows that the **optimal g varies with compute budget, generator-verifier pairing, and task type**. Building on this, they propose **adaptive VG-Search**, which selects g per problem (or per region of generation) using lightweight heuristics. Result: up to +3.1% over beam search, +3.6% over Best-of-N, with **>52% FLOP reduction**.

## Method
- **Step-level signal used**: The question this paper asks is meta-level — *how often* should you invoke any step-level verifier (PRM, score, etc.). The signal driving adaptive granularity is a heuristic over compute budget, problem difficulty estimate, and verifier-vs-generator FLOP ratio.
- **Trigger threshold**: Verifier is invoked every g tokens. Adaptive policy picks g based on (i) remaining compute budget, (ii) task difficulty proxy, (iii) generator-vs-verifier cost ratio. Policy is search/heuristic-based, not learned.
- **Action triggered**: When verifier fires, run beam-style top-k pruning over partial continuations.
- **Training requirement**: **None** for the policy itself; the paper uses off-the-shelf verifier PRMs (e.g., Skywork PRMs at 1.5B and 7B) as the verifier. Adaptive g policy is essentially a tuned heuristic.

## Reported results
- **Up to +3.1% accuracy** over beam search; **up to +3.6%** over Best-of-N.
- **>52% FLOP reduction** at matched accuracy.
- Tested across multiple generator-verifier configurations (varying generator size, varying verifier PRM size: Skywork-PRM 1.5B / 7B), multiple compute budgets, multiple task types (incl. AIME-style competition math).
- Key empirical message: **no single g dominates** — best g moves with budget and task; *adaptive* selection wins.
- Code open-sourced (github.com/hmarkc/VG-Search).

## Why this is HIGH threat
- **Direct quote on niche overlap:** "We make the first attempt toward systematically investigating the impact of verification granularity — that is, how frequently the verifier is invoked during generation."
- This is the exact question CoT-CP claims to answer: *when should the verifier fire?* VG-Search answers it via parameter sweep + adaptive heuristic; CoT-CP would answer it via a CRC objective that picks the firing rate to satisfy a target risk.
- The 52% FLOP reduction is impressive and competitive — CoT-CP must match or beat it while delivering a coverage guarantee VG-Search does not provide.
- Open-source code makes head-to-head comparison feasible *and obligatory*; reviewers will demand it.
- Adaptive VG-Search is a proper baseline — if its heuristic g-policy gets close to CoT-CP's CRC-based g-policy on FLOPs, CoT-CP must lean hard on the calibrated-coverage axis.

## What CoT-CP adds vs this paper
- **Calibration layer that gives P(correct) >= 1-alpha via CRC** — VG-Search's adaptive g is chosen to maximize accuracy / minimize FLOPs *empirically*. There is no risk-control statement: a user cannot say "give me >=90% chance of a correct chain". CoT-CP can.
- VG-Search's adaptive policy selects g but does not gate per-step actions on a calibrated score; CoT-CP combines (a) calibrated per-step gating with (b) calibrated granularity selection.
- VG-Search varies g in token-count units; CoT-CP can express the same axis in **expected-verifier-calls-per-correct-step** units, which is more interpretable to a deployment user with a budget constraint.
- **Required experimental baseline**: **YES, MANDATORY** — VG-Search is the closest non-CP competitor. Use the open-source code with the same Skywork PRMs and matched compute budgets. Report (i) accuracy at matched FLOPs, (ii) achieved coverage at target alpha (CoT-CP only), (iii) FLOPs at matched accuracy.
